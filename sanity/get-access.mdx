A SixthGear administrator must invite you before you can open the CMS. The invitation is connected to one email address, so sign in with that address and the same identity provider used for its Sanity account. The process normally takes only a few minutes. Never share your password with another employee.

<Info>
  You need access to the exact email account that received the Sanity invitation. Keep that inbox open while completing this guide.
</Info>

## Before you begin

Have these items ready:

- Access to the invited email account
- The Sanity invitation email
- A supported, up-to-date browser
- The correct Google or Sanity sign-in method for the invited identity
- Approval from the responsible SixthGear administrator

<Warning>
  Using the wrong Google account or a different sign-in provider can create a separate Sanity identity that does not have SixthGear access. Use the sign-in method already connected to the invited Sanity identity.
</Warning>

<Tip>
  If you use several work and personal accounts in the same browser, compare the displayed email address at every account-selection screen. A private browser window can reduce account-switching confusion.
</Tip>

## Troubleshooting

<AccordionGroup>
  <Accordion title="The invitation email did not arrive">
    - Check **Spam**, **Junk**, **Promotions**, and **Updates**.
    - Search the inbox for `Sanity`.
    - Confirm the invited email address with the administrator.
    - Ask the administrator to review pending invitations and resend the invitation when appropriate.
  </Accordion>

  <Accordion title="Sanity opens the wrong account">
    - Select **Switch accounts**.
    - Sign out of unrelated Sanity sessions.
    - Use a private browser window when account sessions remain mixed.
    - Sign in with the exact account that received the invitation.
  </Accordion>

  <Accordion title="The invitation was accepted using the wrong account">
    - Stop before editing CMS content.
    - Notify the SixthGear administrator.
    - Ask the administrator to remove or revoke the incorrect membership.
    - Send a new invitation to the correct account.
    - Do not share the incorrect account with the intended user.
  </Accordion>

  <Accordion title="The invitation expired">
    - Ask the administrator to review the pending invitation.
    - Revoke the expired invitation if the administrator determines that it is required.
    - Send a new invitation and use only the newest invitation email.
  </Accordion>

  <Accordion title="The user can open the project but cannot edit">
    - Confirm the assigned role. Some roles may be read-only.
    - Ask the administrator to review the user's role and the current Sanity plan.
    - Do not share an administrator account to bypass role limits.

    Role names and availability depend on the verified Sanity project configuration and plan.
  </Accordion>

  <Accordion title="The user can edit but cannot publish">
    - Check the document for field validation errors.
    - Confirm that all required fields are complete.
    - Confirm that the assigned role permits publishing.
    - Ask the administrator to review access if the document is valid but publishing remains unavailable.
  </Accordion>

  <Accordion title="SixthGear CMS is blank or does not load">
    - Refresh the browser once.
    - Wait a few minutes if the invitation was accepted recently.
    - Confirm that the correct account is signed in.
    - Try a private browser window.
    - Temporarily disable browser extensions that may block the Studio.
    - Record the browser, time, URL, and visible error before escalating.
  </Accordion>

  <Accordion title="The project appears, but SixthGear CMS does not">
    - Confirm the production Studio URL with the SixthGear website administrator.
    - Confirm that you opened the correct Sanity project.
    - Confirm the expected dataset where it is visible and safe to review.
    - Verify that the website deployment is online.
    - Escalate the collected details to the SixthGear website administrator.
  </Accordion>
</AccordionGroup>

## Understand your assigned role

Access is based on the role the administrator assigns. A read-only role allows viewing but not editing. A broader role may allow editing, publishing, or project administration, depending on the verified project configuration and Sanity plan.

Administrators should grant only the access required for a person's work. Staff must use named accounts; never share one administrator account among employees.

## Protect account and project information

Never place passwords, API tokens, recovery codes, private keys, full personal email addresses, customer information, billing details, project tokens, Sanity secrets, or private environment values in this guide, a screenshot, or an access request. Blur unrelated inbox messages, account choices, project identifiers, and browser details before a screenshot is approved for publication.
